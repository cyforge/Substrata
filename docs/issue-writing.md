# Inclusive Docs and Issue-Writing Guidance

## Writing Style

- Prefer plain language over internal jargon.
- Expand acronyms the first time they appear unless they are universally known in the repo.
- Keep sentences short when describing tasks, acceptance criteria, and failure modes.
- Use concrete examples when a contributor could interpret the requirement in multiple ways.

## Issue Structure

Implementation issues should include:

- background
- scope
- acceptance criteria
- suggested technologies or references
- dependency notes when work cannot start independently
- contributor fit when the issue is intended for a specific skill level

## Issue Quality Rules

- Keep issues small enough that the reviewer can explain the change in a short paragraph.
- Avoid combining helpers, orchestration, and integration in one task unless the foundation work makes that necessary.
- Do not rely on private notes or local-only planning documents for critical context.
- State the done condition directly in the issue body.

## Contributor Experience

- Write tickets so a contributor from a different geography or work context can still succeed.
- Avoid assumptions about prior blockchain knowledge unless the issue truly requires it.
- When a task is suitable for newcomers, make the first file or entrypoint obvious.
